# Stream Deck Remote

Mobile controller for the Windows `streamdeck-clone` app. The desktop app owns the profiles, buttons, and action execution; this app wakes the PC, connects to the desktop WebSocket server, mirrors the active deck config, and sends button trigger commands.

## Desktop Contract

The Windows app should expose:

- WebSocket host: `0.0.0.0`
- WebSocket port: `37123`
- Incoming messages: `getConfig`, `triggerButton`, `saveConfig`
- Outgoing config message: `{ "type": "config", "config": DeckConfig }`

The mobile editor sends `saveConfig` after button edits/deletes. The desktop Rust/Tauri bridge must handle that message and call its existing config save path for edits to persist on the PC app.

## Native Dev Build

Wake-on-LAN and the phone keyboard need a custom Expo dev client or native build. This repo now includes `expo-dev-client`, an Android native project generated by Expo prebuild, and an EAS development profile.

The generated `android/` folder is now part of this project because Wake-on-LAN and HID keyboard support use native Android modules.

Common Android flow:

```bash
npm run android:dev
npm run start:dev-client
```

Cloud/internal dev build flow:

```bash
eas build --profile development --platform android
npm run start:dev-client
```

## Standalone APK

Use the standalone APK build when you want the phone app to run without Metro,
Expo Go, or a dev server on the PC. This APK still needs the Windows
`streamdeck-clone` app after sign-in because the Windows app owns and executes
the desktop actions.

Cloud APK build:

```bash
npm run build:apk
```

After EAS finishes, install the APK it gives you on the Android phone.

Local APK build on this PC:

```bash
$env:JAVA_HOME="C:\Program Files\Android\Android Studio\jbr"
npm run android:release
```

The local APK is written to:

```text
android/app/build/outputs/apk/release/app-release.apk
```

This local release currently uses the generated debug signing config, so Android
may show it as an app from an unknown source. That is fine for personal testing.
For sharing broadly, create a real release keystore.

## Wake-on-LAN

Wake-on-LAN sends a UDP magic packet to the configured MAC address. It requires:

- Wake-on-LAN enabled in BIOS/UEFI.
- Wake-on-LAN enabled on the Windows network adapter.
- The phone and PC on the same LAN or routed broadcast support.
- The Android native dev build. Expo Go can connect to the desktop app, but it cannot send UDP packets.

The app uses its own Android `WakeOnLan` native module first, then falls back to `react-native-udp` if available.

### Wake-on-LAN Test

1. Install/open the Android native dev build, not Expo Go.
2. On Windows, find the wired or Wi-Fi adapter MAC address with `ipconfig /all`.
3. Enable Wake-on-LAN in BIOS/UEFI and in the Windows network adapter advanced settings.
4. In the app, set MAC address, broadcast address, and WOL port. Start with `255.255.255.255` and port `9`; if your router blocks global broadcast, use your subnet broadcast such as `192.168.1.255`.
5. Put the PC to Sleep first and tap **Start PC** in the app.
6. After Sleep works, test shutdown/hibernate only if your motherboard and adapter support WOL from that power state.

Wake-on-LAN cannot be guaranteed by app code alone; BIOS, adapter power settings, router broadcast behavior, and phone LAN isolation all matter. The app sends the magic packet three times from native Android UDP.

## Windows Sign-In Keyboard

The Android native build includes a manual Bluetooth HID keyboard mode:

1. Open the Android native dev build.
2. Tap **Enable** in the Phone Keyboard panel.
3. Pair the phone with Windows from Bluetooth settings. Windows should see it as a keyboard named `StreamDeck Remote Keyboard`.
4. Wake the PC.
5. Use the keypad in this app to enter your Windows PIN manually.
6. After Windows signs in and the desktop StreamDeck app starts, tap **Connect** in this app.

This does not store the PIN and does not auto-type it. It only sends individual key presses you tap.

Important Android limits:

- Android 9+ is required for the Bluetooth HID Device APIs.
- The phone hardware/ROM must support Bluetooth HID Device mode.
- Android may unregister the HID app if it is not foreground.
- Full reboot reconnect behavior depends on the PC Bluetooth adapter and Windows pairing state.

This app does not store or auto-type the Windows PIN. The Bluetooth keyboard mode works because Windows sees the phone as an external keyboard; the desktop StreamDeck server still cannot control the secure login desktop before sign-in.

### HID Keyboard Test

1. Install/open the Android native dev build, not Expo Go.
2. Tap **Enable** in the Phone Keyboard panel and grant Bluetooth permissions.
3. Tap **Open Android Bluetooth settings** and pair Windows with `StreamDeck Remote Keyboard`.
4. While Windows is unlocked, open Notepad and tap the keypad digits, Backspace, and Enter in the app.
5. Lock Windows with `Win + L`, keep the phone app open in the foreground, and test the keypad at the sign-in screen.
6. After lock-screen typing works, test after Sleep/Wake. Full reboot support depends on whether Windows reconnects the paired Bluetooth keyboard before sign-in.

## Desktop Startup

For automatic availability after sign-in, open the Windows `streamdeck-clone`
app and enable both settings in `General`:

- `Launch on startup`
- `Start minimized to tray`

With those enabled, the normal flow is:

1. Open the standalone APK on the phone.
2. Tap **Start PC**.
3. Tap **Sign In** and use the phone keyboard or Enter button.
4. After Windows signs in, the desktop app starts in the tray/background.
5. Tap **Desktop App** on the phone, connect, sync, and run buttons.

The phone cannot run Windows actions by itself. It sends commands to the desktop
app, and the desktop app executes them on the PC.
